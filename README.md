The project will go through the following phases:

Project Launch Phase 
The launch call is scheduled once the SOW has been signed-off and any other legal considerations taken care of on both ends. We share a launch deck, which has pictures of main stakeholders at ZETAAILABS. The focus of launch call is to introduce the project team to the company as well as get a high-level overview of the project from the company. We do not get into technical details on this call and focus remains on any high-level queries and any dependencies on the Company. 

Project Discovery Phase – Sprint 0 
This is a pivotal phase of the project as it sets the foundation. This foundation helps the later phases be more rapid and agile. The project team consisting of project coordinator and all respective Ticket/Orders get together after the launch call to ensure we have all artefacts needed to start Sprint 0. The focus of this phase is to get clear requirements understanding as laid out in the SOW. The team reaches out to the Company multiple times during this phase to setup calls for better understanding of the requirements. The deliverables from this phase are wireframes and flow of the application. In addition, team starts putting together sprint wise project plan as well as high level architecture document. This phase is also used to onboard the complete team for the project. The duration of this phase is typically for 1-2 weeks depending on complexity of the project.


 

Project Execution Phase – Sprints 1-n 
The design/development sprint is typically for 2-3 weeks. The sprint starts with a sprint planning meeting. The complete team attends the sprint planning meeting. The project coordinator showcases the user stories and the acceptance criteria to all team members. The focus here is for everyone to understand the feature in detail. The developers follow this with creating tasks for each user story and adding estimated time needed to develop it. The testers start writing the test case for each story. The sprint planning takes 3-6 hours based on the size of the project and the number of features being taken in a sprint. The sprint planning is followed with actual development. The team meets everyday at the stand-up where the focus is on what was accomplished the previous day, what is the plan for today and if there are any roadblocks. The developer does a story unit test after finishing development before moving to the testing queue. The testers work in parallel with developers to test each story as it is being developed. Once all stories are developed and tested, the tester focuses on integration testing to ensure the application is working as a whole and nothing is broken from earlier developed and released features. Once the testing team has passed the application, there is an internal showcase with the complete team and the delivery manager. After the internal showcase we do a company showcase where all the features developed in the current sprint are showcased. After the showcase the project coordinator shares the application with the company along with release notes and any known bugs. The team then moves to the next sprint and the same cycle is repeated with another set of features.
 
Project Gold Phase 
After all features have been developed we go into a gold release sprint. In this sprint the focus is on regression testing end-to-end for the complete application and fixing of any bugs. In case of project on multiple platforms this also includes extensive cross-platform testing. At the end of this sprint, we do a showcase of the complete application to the company. This release is labelled as the Gold release on the project plan. 

Project UAT Phase 
With the Gold release we allow for 1-2 weeks of UAT (User Acceptance Testing) by the company. The team on priority will take up any feedback received during this phase. 

Project Warranty Support Phase 
After we have closed the UAT phase we go into the free warranty support phase. The focus of this phase is to support the company with any bugs which may get reported based on more extensive testing being done by the company as well as other extended team members.

OVER ALL TESTING PROCESS 

Test Requirement Gathering 
Test Requirement gathering is the process of identifying and understanding needs of Client/project team. Test Ticket/Order will interact with the client/project team to understand, what are the objectives for the system or product are, what is to be accomplished, how the system or product fits into the needs of the business, and finally, how the system or product is to be used on a day-to-day basis. This would help the testing team in identifying scope for testing. 
Test requirements are captured as first draft of test cases for Application Development / New Media / Gaming projects. In high complexities areas / projects with un-written requirements / testing projects, a feature list is maintained in an excel sheet. 


Test Planning 
Test Plan will be prepared by the Test Ticket/Order as per Test Plan Guidelines. Initial test plan will address system test planning during the SDLC lifecycle. This plan will be updated to cater other testing requirements of the testing phases, like unit, integration and system testing. The test Plan provides testing scope, approach and strategy, milestones for testing. 
The Test Plan will be reviewed and tracked as per the tracking procedure defined for the project. 
An Acceptance plan will be prepared based on client requirements only. The acceptance plan consists of: A suggested schedule for Acceptance Testing by the client/customer 
Hardware, Software and other resources required for Acceptance Testing 
Establish validation methods, procedures, and criteria that will be used to validate the selected products Confirmation of Acceptance criteria 
The Acceptance Plan will be reviewed as defined in PMP based on the Review Procedure and the review records will be maintained properly. 

Preparation of Test Cases and gather test data 
Test cases will be developed by the Test Engineers as per testing levels and types as defined in the test plan. Test cases are prepared using test case template. The inputs for preparation of test cases are the software requirement specifications and/or Design document/model. 
System test cases will be prepared during requirement gathering stage. Integration test cases will be prepared using design documents. 
Test cases will be reviewed as per Review Procedure defined in the Quality Management System. These reviews will be done against requirements and Design to ensure adequacy, completeness, correctness and tractability. 
Acceptance test cases will be prepared and handed over to the client during project inception based on such requirements defined in the Project Management Plan (PMP). 
The acceptance test cases will be executed during deployment, as part of Delivery Procedure. 
Test data will be gathered for executing the test cases according to production environment and operational scenario.
Test Cases Execution 
Unit Testing 
White or black box Unit Testing will be done performed by TEG, only if it is agreed contract with the project / client. Actual tracking of results will be specified in unit test case report. 
As part of NMG application development, unit testing will be performed by developers as self testing. Known defect list will be sent to TEG as part of release note. 
Integration Testing 
The system integrator will compile and link the system in increments. Each increment will need to go through testing of the functionality that has been added, as well as all tests the previous builds went through (regression tests). Within iteration, integration testing will be executed several times until the whole system has been successfully integrated. 
Integration Testing will be performed against the integration test cases. Test case status sheet will be updated for actual results against each test case ad defects will be mapped against each failed test case. Defects are reported in defect tracking tool and are maintained as per defect lifecycle. 
System Testing 
System testing will be initiated through a System release and a Release Note from the Development team to TEG. System testing of the software will be performed against baseline software and the baseline documentation of the requirements specification documentation. 
System Testing is similar to the Integration testing except that it is run under end user simulated environment as far as possible. 
System testing is performed using system test cases. Case status sheet will be updated for actual results against each test case and defects will be mapped against each failed test case. 
Test plan determines the coverage and types of testing to be performed at System Level. Number of System Test cycles is determined by test completion criteria defined in the Test
